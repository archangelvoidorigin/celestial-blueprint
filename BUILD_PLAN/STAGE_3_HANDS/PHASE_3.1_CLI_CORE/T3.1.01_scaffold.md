# TASK T3.1.01 — Scaffold: cli package + pyproject.toml entry point

- **Stage / Phase**: Stage 3 — Hands / Phase 3.1
- **Depends on**: T1.8.01 (Stage 1 complete — backend API built and verified). The CLI does **not** depend on Stage 2.
- **Produces**: `backend/cli/__init__.py`, `backend/cli/completions/.gitkeep`, edit to `backend/pyproject.toml`

## Context

The Celestial Blueprint backend (`backend/`) already has a `pyproject.toml` (created in T1.1.02) with `httpx` listed under `[project.optional-dependencies] dev`. The Stage 3 CLI is a Python package at `backend/cli/` installed into the same venv. It needs `httpx` as a **runtime** dependency (not dev-only), and the `cb` entry point must be registered in `[project.scripts]` so `pip install -e .` creates the `cb` binary.

## Objective

Create the `cli` Python package skeleton and update `pyproject.toml` to register the `cb` entry point and promote `httpx` to a runtime dependency.

## Instructions

1. Create `backend/cli/__init__.py` with exactly:
   ```python
   """Celestial Blueprint CLI — thin HTTP wrapper over the REST API."""
   ```

2. Create `backend/cli/completions/.gitkeep` (empty file — ensures the directory is tracked by git).

3. Edit `backend/pyproject.toml`. The current file looks like this (from T1.1.02):
   ```toml
   [project]
   name = "celestial-backend"
   version = "0.1.0"
   description = "Celestial Blueprint planning backend (API-first)"
   requires-python = ">=3.11"
   dependencies = [
       "fastapi>=0.115,<0.116",
       "uvicorn[standard]>=0.32,<0.33",
       "pydantic>=2,<3",
       "pyyaml>=6,<7",
   ]

   [project.optional-dependencies]
   dev = ["pytest>=8,<9", "httpx>=0.27,<0.28"]

   [tool.pytest.ini_options]
   testpaths = ["tests"]
   pythonpath = ["."]
   ```

   Replace it with exactly:
   ```toml
   [build-system]
   requires = ["setuptools>=61"]
   build-backend = "setuptools.build_meta"

   [project]
   name = "celestial-backend"
   version = "0.1.0"
   description = "Celestial Blueprint planning backend + CLI (API-first)"
   requires-python = ">=3.11"
   dependencies = [
       "fastapi>=0.115,<0.116",
       "uvicorn[standard]>=0.32,<0.33",
       "pydantic>=2,<3",
       "pyyaml>=6,<7",
       "httpx>=0.27,<0.28",
   ]

   [project.optional-dependencies]
   dev = ["pytest>=8,<9"]

   [project.scripts]
   cb = "cli.main:main"

   [tool.setuptools]
   packages = ["app", "cli"]

   [tool.pytest.ini_options]
   testpaths = ["tests"]
   pythonpath = ["."]
   ```

   Key changes: `httpx` moved from `dev` to main `dependencies`; `[project.scripts]` section added; `[build-system]` and `[tool.setuptools]` added.

   > **Why `[build-system]` + `[tool.setuptools]` are required:** the `backend/` directory has multiple top-level packages (`app/`, `cli/`, and `tests/`). When `pip install -e .` runs, setuptools' automatic flat-layout discovery refuses to proceed and fails with `error: Multiple top-level packages discovered in a flat-layout: ['app', 'cli']`. Declaring `packages = ["app", "cli"]` explicitly tells setuptools exactly what to install (and excludes `tests/`), and the `[build-system]` table pins the build backend. Without these two tables the install — and therefore the `cb` entry point — does not get created.

4. Re-install the package into the venv so the entry point is available:
   ```bash
   cd /home/void-yggdrasil/CHAOS_DAO_FORGE/00_CORTEX/01_CREATION_REALM/Celestial_Blueprint/backend
   source .venv/bin/activate 2>/dev/null
   pip install -e . -q
   ```
