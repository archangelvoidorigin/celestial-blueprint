# TASK T6.3.02 — `ChroniclePanel.tsx` — Web Notification after CYCLE chronicle

- **Stage / Phase**: Stage 6 — Ascension / Phase 6.3
- **Depends on**: T1.6.07 (`ChroniclePanel.tsx` exists), T6.3.01
- **Produces**: edit to `frontend/src/components/chronicle/ChroniclePanel.tsx`

## Context

`ChroniclePanel.tsx` (T1.6.07) is the canvas drawer that lists chronicles and lets the user log RAW notes. Important: the panel itself only ever POSTs `kind: "raw"` chronicles — CYCLE chronicles are written by the methodology SKILL / `cb` CLI at the end of a planning session, not from this UI. So the notification cannot hang off a POST in this component. Instead it fires when a **new CYCLE chronicle appears in the fetched chronicle list** (the panel already calls `listChronicles` on open and after each add). This works regardless of whether the cycle was created by the CLI, the skill, or another client.

The notification shows the blueprint id and the "Next session:" line extracted from the cycle body (same logic as `tools/notify.sh`). Web Notifications require permission, requested once at mount. If permission is denied or unavailable, it degrades silently. The first list-sync only records a baseline (the newest existing cycle id) so pre-existing cycles don't fire a notification on load — only cycles that arrive *after* mount notify.

## Objective

Edit `ChroniclePanel.tsx` to request notification permission on mount and fire a Web Notification when a newly-arrived CYCLE chronicle shows up in the fetched list.

## Instructions

1. Edit `frontend/src/components/chronicle/ChroniclePanel.tsx`:

   a. The component already imports `useEffect, useState` from React (T1.6.07). Add `useRef` to that import:
      ```tsx
      import React, { useEffect, useRef, useState } from "react";
      ```

   b. Inside the component body (alongside the existing `useState` hooks), add a ref that tracks the last cycle id we've already accounted for:
      ```tsx
      const lastCycleId = useRef<number | null>(null);
      ```

   c. Add a `useEffect` that requests notification permission once on mount:
      ```tsx
      useEffect(() => {
        if ("Notification" in window && Notification.permission === "default") {
          Notification.requestPermission();
        }
      }, []);
      ```

   d. Add a second `useEffect`, keyed on `chronicles`, that fires a notification when a newer CYCLE appears than any seen before:
      ```tsx
      useEffect(() => {
        const cycles = chronicles.filter((c) => c.kind === "cycle");
        if (cycles.length === 0) return;
        const newestId = Math.max(...cycles.map((c) => c.id));
        // First sync establishes a baseline — don't notify for pre-existing cycles.
        if (lastCycleId.current === null) {
          lastCycleId.current = newestId;
          return;
        }
        if (newestId <= lastCycleId.current) return;
        lastCycleId.current = newestId;
        if (!("Notification" in window) || Notification.permission !== "granted") return;
        const newest = cycles.find((c) => c.id === newestId)!;
        const nextLine = newest.body
          .split("\n")
          .find((l) => l.trim().toLowerCase().startsWith("next session:"))
          ?.split(":")[1]
          ?.trim();
        new Notification(`Celestial Blueprint — ${blueprintId}`, {
          body: nextLine ? `Next: ${nextLine}` : "Planning cycle complete.",
          icon: "/favicon.ico",
        });
      }, [chronicles, blueprintId]);
      ```

   These additions reference only variables that already exist in T1.6.07's `ChroniclePanel`: the `chronicles` state array (whose items have `id`, `kind`, `body`) and the `blueprintId` prop.
