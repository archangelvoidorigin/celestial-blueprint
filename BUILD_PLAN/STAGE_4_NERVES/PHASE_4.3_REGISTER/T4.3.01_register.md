The new `celestial-blueprint` entry uses the env-wrapper bash pattern (same as `firecrawl`, `github`, `context7` in that file): a bash one-liner that sources `backend/.env` before exec. It uses `backend/.venv/bin/python3` (not system `python3`) so httpx is guaranteed available.

**About `backend/.env`:** T1.1.03 creates `backend/.env.example` only ‚Äî `.env` itself is git-ignored and is NOT created by any task. The user creates it by copying `.env.example` to `.env` and filling in `CELESTIAL_AUTH_TOKEN` during backend setup (it is required to run the backend, which is a Stage 4 prerequisite). To avoid a hard launch failure if `.env` is missing, the bash wrapper below sources it **only if it exists** (`[ -f ‚Ä¶ ] && . ‚Ä¶`) rather than an unconditional `source` that aborts the whole command. If `.env` is absent the server still starts; it just sends no auth header (the backend will return a clear 401, which surfaces as an MCP error ‚Äî far better than the MCP server silently failing to launch). `CELESTIAL_API_URL` defaults to `http://localhost:8787` in server.py if not set. The `CELESTIAL_AUTH_TOKEN` the MCP server uses must match the one the running backend was started with.
